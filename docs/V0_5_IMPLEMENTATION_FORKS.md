# SoulForge v0.5 第二批实现分叉裁定

本文记录 v0.5 在产品目标确定后，进一步细化的实现分叉。

排序规则：

- A：最低完成度 / 最保守实现
- B：可用级
- C：产品级
- D：终局级 / 最大完成度

## 已裁定

### 31. v0.5 完成定义

选择：D。

v0.5 目标按完整产品级超级编辑器定义，不以小范围 demo 为最终目标。

若后续出现冲突，以后续更高编号裁定为准。

### 32. 资源模式成熟度

选择：D。

各资源模式最终目标均达到独立专业工具水平。

### 33. action

选择：D。

目标：动作级完整编辑体验。

### 34. chr

选择：特殊裁定。

chr 重点不是模型编辑，而是各个词条属性、起始帧、结束帧等行为属性编辑。

### 35. menu

选择：C。

目标：菜单文本、布局参数、图标引用等结构化编辑。

### 36. obj

选择：C。

目标：对象引用、结构化参数和替换能力。

### 37. script

选择：D。

目标：完整语言服务级能力，包括 AI 修改、分析、重构方向。

### 38. sfx

选择：C。

目标：特效参数结构化编辑和替换。

### 39. other

选择：B。

目标：先分类、识别和基础处理。

### 40. 图片 / 纹理

选择：C。

目标：可预览、替换、导出和重新打包，不要求内置完整图像编辑器。

### 41. raw edit

选择：D。

raw edit 必须具备 hex、byte diff、patch、validator 等完整安全链。

### 42. hex editor

选择：D。

目标：完整 hex 工作流。

### 45. BND 写回

选择：D。

目标：新增、删除、重命名、重排 child，并完整重打包。

### 48. overlay 模型

选择：参考 Smithbox。

保留 base + overlay 思路，并参考 Smithbox 在资源组织上的成熟方式。

### 49. base 缺失

选择：参考建议。

默认采用：没有 base 也能打开 Mod，但能力降级；同时自动寻找可能安装位置。

### 50. game profile

选择：D。

目标：profile 插件化，社区可扩展游戏支持。

### 60. ParamDef

选择：D。

目标：paramdef editor、版本管理、schema diff。

### 61. EMEVD instruction schema

选择：D。

目标：完整 instruction database、用户扩展、版本化。

### 62. EMEVD 编辑方式

选择：D。

目标：图形化事件流程编辑器 + DSL/IR + raw view 三视图同步。

### 64. 坐标编辑

选择：C。

目标：可视化 gizmo 和空间编辑体验。

### 72. 工具协议

选择：C。

目标：MCP-compatible schema，先内部使用。

### 75. 后台任务系统

选择：D。

目标：完整 job system：依赖图、失败重试、持久化、恢复。

### 76. Bridge 模型

选择：D。

目标：Bridge daemon + request protocol + cancellation + streaming result。

### 80. CI

选择：不做，但保留可能性。

当前不纳入 v0.5 必做范围，但未来可恢复为跨平台 CI。

### 84. Blender MCP

选择：B。

只保留文档预留，不实现。

### 85. 本地 LLM

选择：B。

只保留可能性，不实现 runtime。

### 86. Vector DB / Embedding

选择：B。

保持非目标，只保留未来可能性。

### 93. 多窗口 / 多工程

选择：D。

目标：多工程 workspace，可跨工程比较和迁移 patch。

### 97. 用户注释

选择：D。

目标：注释进入 AI context、patch bundle 和协作流程。

### 98. 团队协作

选择：C。

目标：patch bundle 分享，不立即做多人实时协作。

### 99. Git 集成

选择：D。

目标：完整 Git 集成：diff、commit、branch、restore。

## 未特别裁定项

其余未列项目采用上一版建议：

- 31 之后未特别说明的项目默认采用推荐方案；
- 后续编号优先级高于前面编号；
- 如果新裁定与旧裁定冲突，以后发布的决策为准。

## 实现提醒

这些目标属于产品方向，不代表所有能力必须一次完成。

Codex 仍只执行窄实现任务，架构边界和产品范围必须先裁定。
