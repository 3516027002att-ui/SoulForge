# v0.3 Synthetic Event and PARAM Fixture Layouts

This document defines two tiny SoulForge-owned binary fixture layouts for v0.3 parser plumbing work.

They are not FromSoftware native EMEVD or PARAM specifications.

The purpose is to prove Bridge export routing, structured symbol shapes, reference-role metadata, and confidence labeling without committing real game assets.

## Common rules

- All integer fields are little-endian signed 32-bit values.
- These fixtures may be generated by smoke scripts in temporary directories.
- They must never be treated as native game-format authority.
- Confirmed synthetic fixture output should use high-confidence symbol metadata, but diagnostics must clearly say that native authority is false.

## Synthetic PARAM fixture

Magic and marker:

```text
0x00  4 bytes   ASCII magic: PARA
0x04  4 bytes   ASCII marker: SFPR
```

Header:

```text
0x08  int32     version, currently 1
0x0C  int32     row count
0x10  int32     row table start offset
0x14  int32     string pool start offset
```

Row table, repeated `row count` times:

```text
int32 rowId
int32 rowNameStringPoolRelativeUtf16Offset
int32 value
int32 typeCode
```

`typeCode` values:

```text
1 = int32
2 = bool32
other = unknown
```

String pool:

```text
UTF-16LE null-terminated row names
```

Expected Bridge output:

```text
resourceKind = param
parseStatus = partial
diagnostic code = PARAM_SYNTHETIC_FIXTURE_CONFIRMED
```

Each row should export:

```text
ParamExport.rows[].rowId
ParamExport.rows[].rowName
ParamExport.rows[].fields[0].name = value
ParamExport.rows[].fields[0].type
ParamExport.rows[].fields[0].value
raw.confidence = high
raw.nativeFormatAuthority = false
```

## Synthetic event fixture

Magic and marker:

```text
0x00  4 bytes   ASCII magic: EVD\0
0x04  4 bytes   ASCII marker: SFEV
```

Header:

```text
0x08  int32     version, currently 1
0x0C  int32     event count
0x10  int32     event table start offset
```

Event row, repeated `event count` times:

```text
int32 eventId
int32 instructionCount
int32 instructionTableStart
int32 reserved
```

Instruction row, repeated `instructionCount` times:

```text
int32 instructionIndex
int32 opcode
int32 roleCode
int32 argValue
int32 paramNameCode
int32 reserved
```

`roleCode` values:

```text
0 = unknown
1 = flag
2 = eventId
3 = entityId
4 = regionId
5 = paramId
6 = textId
```

`paramNameCode` values:

```text
0 = none
1 = SpEffectParam
2 = EquipParamGoods
3 = NpcParam
```

Expected Bridge output:

```text
resourceKind = event
parseStatus = partial
diagnostic code = EMEVD_SYNTHETIC_FIXTURE_CONFIRMED
```

Each event should export:

```text
EventExport.events[].eventId
EventExport.events[].instructions[].index
EventExport.events[].instructions[].name
EventExport.events[].instructions[].args[0].role
EventExport.events[].instructions[].args[0].value
EventExport.events[].instructions[].args[0].confidence = high
raw.confidence = high
raw.nativeFormatAuthority = false
```

## Wire-up status

`SyntheticFixtureExports.cs` defines the parsing/export helpers for these layouts.

The intended route is:

```text
Program.cs export-event/export-param
  -> SemanticCandidateExports.TryExport
    -> SyntheticFixtureExports.TryExport first
    -> low-confidence native candidate fallback second
```

If GitHub write filtering blocks direct edits to the existing router files, Codex should perform this final wire-up locally and run the smoke scripts after build.
